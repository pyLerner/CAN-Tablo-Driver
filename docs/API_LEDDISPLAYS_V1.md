# REST API: LED displays (CAN) — версия 1

> **Устарело для текущего кода.** Актуальный контракт HTTP и конфигурации: **[API_LEDDISPLAYS_V2.md](API_LEDDISPLAYS_V2.md)** (одно табло, `color-map`, зоны `[display.N]`, `PUT /api/leddisplays/v1/values/update`). Ниже сохранено описание прежней схемы мультитабло и эндпоинтов `route/set`, `tickerboard/set` для истории.

Документ описывает HTTP-интерфейс сервиса после реализации плана «REST API и мультитабло для CAN LED» (общий `Bus`, секции табло в TOML, FastAPI + uvicorn). Все пути относительны к базовому URL сервера (например `http://127.0.0.1:8000`).

## Общие соглашения

| Параметр | Значение |
|----------|----------|
| Формат тела запросов | `application/json` |
| Формат ответов | `application/json` |
| Кодировка | UTF-8 |
| Версия API | префикс пути `/api/leddisplays/v1/` |

Ключи в JSON в теле запросов и ответах — **kebab-case** (`first-string`, `route-width`), как в ТЗ. Внутреннее состояние и совместимость с существующим JSON для файла могут использовать `firstString` и т.д.; HTTP-слой выполняет маппинг.

Отправка на шину CAN выполняется **асинхронно** относительно обработчика HTTP (например `asyncio.create_task` + выполнение блокирующего кода в пуле потоков), чтобы не блокировать event loop.

## `GET /api/ping`

Проверка живости сервиса.

### Ответ `200 OK`

```json
{
  "running": "OK",
  "timestamp-utc": "2026-03-29T12:34:56.789012+00:00"
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `running` | string | Константа успешного состояния (`"OK"`). |
| `timestamp-utc` | string | Метка времени в UTC, формат ISO 8601 с суффиксом смещения `+00:00` (или эквивалент, согласованный с потребителями). |

## `POST /api/leddisplays/v1/config/set`

Частичное обновление конфигурации табло: валидация, **слияние** с текущим состоянием в памяти и с файлом `src/config.toml`, затем перезагрузка in-memory конфигурации.

- Отсутствующие в теле запроса поля **не затирают** ранее сохранённые значения ни в файле, ни в памяти.
- После успешной записи конфиг применяется без перезапуска процесса (перечитанный/слитый конфиг).

### Тело запроса

Структура с **опциональными** вложенными блоками (Pydantic-модель). Ниже — логическая схема; точный набор полей соответствует секциям TOML и общим параметрам шины.

**Общие параметры шины** (аналог секции `[can]` и при необходимости `[iso-tp]`, `[logs]`):

| Поле (пример) | Тип | Описание |
|---------------|-----|----------|
| `channel` | string | Интерфейс CAN, например `can0`. |
| `bitrate` | integer | Скорость шины в бит/с. |

**Секции табло** (каждая опциональна; имена соответствуют секциям конфига):

- `front-display`
- `side-front-display`
- `side-rear-display`
- `rear-display`
- `ticker-board` — одна секция таблицы или **массив** объектов (несколько `[[ticker-board]]` в TOML).

В каждой секции, помимо идентификаторов ISO-TP, задаётся геометрия и поведение, аналогично нынешнему `[tabloRouteTwoStrings]`:

| Поле | Тип | Описание |
|------|-----|----------|
| `sender-tx-id` | integer | CAN ID передачи (отправитель → табло). |
| `sender-rx-id` | integer | CAN ID приёма (ответы табло → отправитель). |
| `route-width` | integer | Ширина колонки маршрута в пикселях/логических единицах разметки (в ТЗ встречалась опечатка `route-width:`; в API используется ключ **`route-width`**). |
| `route-text-scale-x` | number | Масштаб текста маршрута по горизонтали. |
| `width`, `height` | integer | Размеры области табло. |
| `pad-left`, `pad-right`, `pad-top`, `pad-bottom` | integer | Отступы. |
| `right-lines` | integer | `1` или `2` — одна или две строки справа от маршрута (для переднего/бокового режима). |

Для бегущей строки (`ticker-board`) колонка маршрута может отсутствовать или иметь нулевую ширину; детали — в конфиге секции (одна или две горизонтальные полосы).

**Цвет текста** (глобально или с перекрытием в секции — по реализации; достаточно глобального значения в памяти):

| Поле | Тип | Описание |
|------|-----|----------|
| `color` | string | Например `"YELLOW"` → внутренний маппинг на константы `COLOR_*` при рендере. |

Пример **фрагмента** тела (не исчерпывающий список полей):

```json
{
  "channel": "can0",
  "bitrate": 500000,
  "front-display": {
    "sender-tx-id": 2016,
    "sender-rx-id": 2024,
    "route-width": 120,
    "right-lines": 2
  },
  "ticker-board": {
    "sender-tx-id": 2032,
    "sender-rx-id": 2040
  },
  "color": "YELLOW"
}
```

Несколько тикеров в одном запросе — массив блоков:

```json
{
  "ticker-board": [
    { "sender-tx-id": 2032, "sender-rx-id": 2040, "width": 256, "height": 32 },
    { "sender-tx-id": 2033, "sender-rx-id": 2041, "width": 128, "height": 32 }
  ]
}
```

### Ответы

| Код | Описание |
|-----|----------|
| `200` | Конфиг принят, слит, записан в TOML, состояние обновлено. |
| `422` | Ошибка валидации (неверные типы, недопустимые размеры и т.д.). |
| `500` | Внутренняя ошибка (запись файла, применение конфига). |

Тело ответа при успехе может быть пустым или содержать краткое подтверждение — по реализации сервиса; клиентам достаточно кода `200`.

## `POST /api/leddisplays/v1/route/set`

Сохраняет в памяти три строки маршрута/табло и **асинхронно** отправляет отображаемое содержимое на все табло типа «маршрут» (переднее, боковые, заднее), согласно активным секциям конфигурации. **Тикер не изменяется и не затрагивается этим запросом.**

### Тело запроса

| Поле | Тип | Обязательность | Описание |
|------|-----|----------------|----------|
| `first-string` | string | да* | Первая строка (маршрут / основная зона в зависимости от табло). |
| `second-string` | string | да* | Вторая строка. |
| `third-string` | string | да* | Третья строка. |

\* Уточните в реализации: либо все три обязательны, либо допускаются пустые строки при опциональных полях.

```json
{
  "first-string": "Маршрут 42",
  "second-string": "Остановка",
  "third-string": "Дополнительно"
}
```

### Ответы

| Код | Описание |
|-----|----------|
| `200` | Строки сохранены; отправка на шину поставлена в очередь/выполняется в фоне. |
| `422` | Ошибка валидации. |

---

## `GET /api/leddisplays/v1/route/set`

Возвращает **текущие** строки маршрута из памяти (последнее успешное состояние после `POST` или начальная загрузка).

### Ответ `200 OK`

```json
{
  "first-string": "…",
  "second-string": "…",
  "third-string": "…"
}
```

Метод и путь совпадают с `POST`, различается только HTTP-метод — как в ТЗ.

---

## `POST /api/leddisplays/v1/tickerboard/set`

Сохраняет в памяти строки внутрисалонного тикера и **асинхронно** отправляет данные **только** на табло секций `ticker-board` (все настроенные в конфиге экземпляры). Маршрутные табло не затрагиваются.

### Тело запроса

| Поле | Тип | Описание |
|------|-----|----------|
| `first-string` | string | Первая бегущая строка / верхний блок. |
| `second-string` | string | Вторая строка / нижний блок (если конфиг тикера предусматривает два блока). |

```json
{
  "first-string": "Текст тикера 1",
  "second-string": "Текст тикера 2"
}
```

### Ответы

| Код | Описание |
|-----|----------|
| `200` | Сохранено; отправка на тикер — в фоне. |
| `422` | Ошибка валидации. |

## Поведение относительно файла текста и CLI

- При работе **сервиса с API** приоритет источника текста для отправки — **состояние из API** (последние `POST` маршрута и тикера).
- Файл из `[TextIn]` / `text-in.json` может использоваться для офлайн-режима или режима без HTTP — по политике развёртывания.

## Запуск сервиса

Отдельная точка входа (например `python -m … api` или `src/api_server.py`) с **uvicorn**; существующие режимы `send` / `recv` / `demo` CLI остаются доступными параллельно.

## Краткая таблица эндпоинтов

| Метод | Путь | Назначение |
|-------|------|------------|
| `GET` | `/api/ping` | Liveness, UTC timestamp |

| `POST` | `/api/leddisplays/v1/tickerboard/set` | Две строки тикера → память + все секции ticker-board |

## Примечание для интеграторов

Реальная шина CAN и несколько ISO-TP стеков на одном `Bus` не требуют от клиента HTTP никаких особых действий: ограничения сняты на стороне сервиса. Долгие передачи не блокируют ответы API благодаря фоновой отправке.
